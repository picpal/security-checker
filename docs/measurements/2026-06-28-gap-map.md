## 핵심 발견

1. **"도구 미활용" 가설 기각.** spotbugs 는 gradle 9.5.1 로 **빌드 성공·실행**됐다
   (rule_id: `CRLF_INJECTION_LOGS` 163, `URF_UNREAD_PUBLIC_OR_PROTECTED_FIELD` 110,
   `SPRING_ENDPOINT` 28, `EI_EXPOSE_REP2` 25…, 총 398건). 그런데도 CWE-476/754/404
   는 0건. 도구를 안 써서가 아니라 **spotbugs/semgrep 가 못 잡는 범주**다.

2. **CWE-476/754/404 의 정체 = IDE 인스펙션 영역.** 수정 diff 가
   `catch (Exception e)` → `catch (DataAccessException e)` 처럼 **광범위 catch 를 구체
   예외로 좁힌 것**(coderay/IntelliJ 의 "overly broad catch" 인스펙션). null 가드·
   리소스도 동류의 방어 코딩/코드 스멜. 바이트코드 보안(spotbugs)·패턴 보안(semgrep)
   둘 다의 영역 밖, coderay 의 고유 강점.

3. **양방향 갭.**
   - coderay > secscan: CWE-476/754/404 (broad catch 등 코드 스멜).
   - secscan deep > coderay: CRLF/expose 류. **단** `CRLF_INJECTION_LOGS` 163 은
     findsecbugs 의 고FP 룰(모든 `log.x(userInput)` 발화) — 신뢰도 계층화(B)가
     **우선 0 · 검토후보 355** 로 강등해 정확도를 보존했다(설계대로 작동).
   - spotbugs 는 Java SQL 빌더 `CWE-89` 2건도 탐지하지만, **#305 의 MyBatis XML
     `${}` 는 바이트코드 밖**이라 미탐 — #305 가 여전히 커스텀 룰 영역인 이유.

## D(커스텀 룰셋) 범위 확정

갭 지도가 D 의 범위를 정확히 가른다:

- **포함(저FP·고가치, 즉시 룰화)**: #305 MyBatis `${}` SQLi · CWE-259 하드코딩 ·
  CWE-760 zero salt. 모두 **맥락 무관하게 단정 가능한 안티패턴** → semgrep 커스텀
  룰로 낮은 FP 탐지 가능.
- **보류(고FP·IDE 영역)**: CWE-476/754/404. broad-catch 는 `catch (Exception` 패턴
  으로 잡히지만 *의도적 swallow 가 정당한 경우*(위 AuthService 처럼 인증 흐름 비차단)
  가 많아 FP 폭발. 정확도 우선(원칙1)상 커스텀 룰로 밀어붙이지 않는다 — IDE
  인스펙션(coderay)·향후 전용 linter 통합 영역으로 남긴다.

## 부수 발견

- **spotbugs 어댑터의 gradlew 미지원**: `_default_build` 가 시스템 `gradle compileJava`
  를 호출한다(`secscan/adapters/spotbugs.py:34`). 이번엔 시스템 gradle 9.5.1 이 있어
  성공했지만, wrapper 만 있고 버전 고정인 프로젝트에선 빌드 실패→SKIPPED 가능.
  **개선 후보**: 프로젝트에 `gradlew` 가 있으면 `./gradlew` 우선 사용.
- deep 의 recall↑/precision↓ 가 실측됨(398건, 검토후보 355). 신뢰도 계층화가 노이즈를
  우선순위에서 분리해 "정확도 우선" 원칙을 지킴.

## 결론

coderay 격차는 **두 갈래**다 — (1) injection/secret/crypto 안티패턴은 **커스텀 룰(D)**
로 메운다(3종 확정), (2) broad-catch 등 **코드 스멜은 IDE 인스펙션 고유 영역**이라
secscan 의 패턴/바이트코드 SAST 로는 정확히 못 메운다(억지 룰화는 FP 역효과).
**C(함수간 taint)는 이 갭 어디에도 해당 없음**(PoC 결론 재확인).
