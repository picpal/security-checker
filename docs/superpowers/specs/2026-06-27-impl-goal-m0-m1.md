# 구현 Goal — JVM 보안 점검 하이브리드 도구 (M0 + M1)

> 이 문서는 자율 구현 에이전트에게 줄 **작업 지시(goal)**다. 범위는 spec의 **M0(doctor) + M1(정확한 SCA 수직 슬라이스)** 까지. 나머지 마일스톤(M2~M5)은 후속 goal로 분리한다.

---

## 목표

JVM(Java/Kotlin) 프로젝트용 **보안 점검 하이브리드 CLI 도구**를 spec대로 구현한다. 이번 goal의 산출물은 "정확한 SCA 수직 슬라이스" — 의존성 취약점을 탐지하고, **도달성(reachability) 분석으로 노이즈를 걷어낸 뒤**, typed finding → SARIF + Markdown 보고서까지 흐르는 한 줄기.

## 필독 (시작 전 반드시)

구현 전에 다음 spec을 끝까지 읽고 그대로 따른다. 아래 요약과 spec이 충돌하면 **spec이 우선**:

```
docs/superpowers/specs/2026-06-26-jvm-security-scanner-design.md
```

## 핵심 원칙 (위반 금지)

1. **정확도 우선** — FP를 줄이는 레버를 모든 것에 우선한다.
2. **결정적/LLM 경계** — 스캐너 실행·정규화·도달성은 결정적 코드. Claude는 증거 해석자일 뿐, 판정 oracle이 아니다.
3. **내부는 typed finding 모델, SARIF는 출력 전용** — SARIF를 내부 canonical로 쓰지 말 것(SCA가 깨진다). spec 6절 데이터 모델을 따른다.
4. **자동 억제 금지** — 억제는 사람이 확정. spec 12절.
5. **부분 실패는 정상** — 한 스캐너가 죽거나 도달성이 타임아웃해도 나머지 결과는 유효하게 보고(partial-result). "전부 성공" 위장 금지.
6. **구현 언어 Python**, 어댑터 인터페이스 추상화(새 스캐너 = 어댑터 1개 추가, 코어 무변경).

## 이번 범위

### M0 — `doctor`
- `Trivy`, `osv-scanner`, `dep-scan`/`atom` 설치 여부 + 런타임(`Java 21`, `Node`, 가용 메모리) 점검 CLI.
- 누락/부적합 시 구체적 설치·해결 가이드 출력.
- 어댑터 스모크 테스트의 기반.

### M1 — 정확한 SCA 수직 슬라이스
1. **스캐너 어댑터**: Trivy + osv-scanner. 병렬 실행, `status`(ok/failed/timeout/skipped), 자원 거버넌스(timeout/메모리/cache-dir) — spec 5.1·11절.
2. **정규화 → typed finding 모델**: 원시 출력 → `Finding`(spec 6절). 결정적. **골든 테스트 1순위**(샘플 원시출력 → 기대 findings).
3. **합의 dedup**: Trivy ∪ osv, 동일 컴포넌트/advisory 병합 + consensus 스코어.
4. **도달성 엔진**: dep-scan/atom으로 SCA findings에 도달성 증거 주입. **4중 안전장치 필수**(spec 10.1):
   - 크기 인지 자동(임계 이하 자동 풀가동, 초과 시 확인)
   - atom 슬라이스 코드 해시 캐싱
   - max-memory/timeout 예산 + 초과 시 도달성만 스킵하고 `reachability=unknown` 폴백
   - doctor 사전 점검 연계
5. **출력**: typed findings(Claude용) + SARIF 2.1.0 + Markdown 보고서.
6. **취약 픽스처 프로젝트**: 알려진 CVE 의존성 + **도달 가능/불가능 케이스를 의도적으로 배치**한 소형 JVM 프로젝트.
7. **도달성 정확도 실측(M1의 존재 이유)**: 도달성 on/off를 비교해 **FP가 실제로 줄어드는지 precision/recall + 비용(시간·메모리)을 수치로 측정**한다.

## 작업 방식

- **TDD** — 정규화 레이어부터 골든 테스트로 시작.
- 작은 단위로 커밋, 마일스톤 검증 게이트 통과 후 다음으로.
- Maven/Gradle 해석 실패모드(멀티모듈·BOM·lockfile 부재·shaded JAR)를 의식하고, 실패 시 partial + 명시 경고.

## 하지 말 것

- 백로그 손대기: IaC/DAST/CI/주기점검/패치 모니터링/Opengrep/CodeQL/LSP.
- Claude 자동 억제.
- 상용·계정 종속 툴(Snyk, ggshield 등).
- SARIF를 내부 모델로 사용.
- TruffleHog 자동 온라인 검증(M2 범위이며 opt-in).

## 완료 기준 (이번 goal)

- [ ] `doctor`가 스캐너·런타임·메모리를 정확히 진단하고 누락 가이드를 출력한다.
- [ ] `scan --profile accurate-sca --target <픽스처>` 가 typed findings → SARIF + Markdown을 산출한다.
- [ ] 픽스처의 **도달 불가 취약점이 `reachability=unreachable`로 표기**되고, 도달성 on/off 비교로 **FP 감소가 수치로 측정**된다.
- [ ] 도달성이 예산 초과 시 `reachability=unknown`으로 폴백하고 탐지 결과는 유지된다(partial-result).
- [ ] 정규화 골든 테스트 + 부분실패 테스트가 green.
- [ ] 도달성 정확도/비용 실측 결과를 짧은 리포트로 남긴다(이후 투자 판단 근거).
