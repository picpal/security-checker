# JVM 보안 점검 하이브리드 Agent — 설계 문서

- **작성일**: 2026-06-26
- **상태**: 설계 합의 완료 (구현 plan 작성 직전)
- **타겟**: JVM(Java/Kotlin) 프로젝트
- **실행 형태**: 하이브리드 (결정적 오케스트레이터 + Claude 해석/보고서 레이어)
- **실행 환경**: 로컬 개발 머신, 개인 사용 우선

---

## 1. 목적

오픈소스 보안 스캐너들을 **요청한 수준·범위에 맞게 병렬 실행**하고, 결과를 **하나의 정규화된 형식으로 종합**한 뒤, **Claude가 트리아지·도달성 판정·보고서 작성**까지 수행하는 보안 점검 도구를 만든다.

핵심 가치: 개별 스캐너는 노이즈가 많고 출력 포맷이 제각각이다. 이 도구는 (1) 실행을 결정적·재현가능하게 캡슐화하고, (2) 출력을 SARIF로 통일하며, (3) "수천 findings 중 무엇이 진짜 위험한가"를 LLM으로 판별한다.

---

## 2. 범위

### MVP (이번 사이클)
- **정적 3종**: SAST + SCA(의존성) + Secret
- 로컬 실행, 단일 JVM 프로젝트 대상
- 통합 SARIF + 사람이 읽는 Markdown 보고서 출력

### 백로그 (이후 사이클, 각자 별도 spec)
- IaC/컨테이너 점검 (`+iac` 애드온: Trivy/Checkov)
- DAST (`+dast` 애드온: ZAP/Nuclei — 구동 중 앱 필요)
- CI 파이프라인 통합 (PR 게이트)
- 주기 점검 (launchd/cron + 결과 diff)
- 신규 라이브러리 패치 모니터링 (Dependency-Track 등 상주 플랫폼)
- 깊은 SAST: Opengrep(cross-function taint), CodeQL(라이선스 허용 시)
- LSP 기반 코드 질의(Serena류)로 도달성 정밀화

### 명시적 비범위 (Out of Scope)
- 상주 서버/대시보드 (SonarQube, Dependency-Track 같은 플랫폼형)
- 상용/계정 종속 도구 (Snyk, ggshield 등)
- 자동 수정(auto-fix) — MVP는 탐지·보고까지만

## 3. 아키텍처

경계를 **SARIF**로 긋는 4계층 구조. 결정적 영역과 LLM 영역을 SARIF 인터페이스로 분리한다.

```
┌─────────────────────────────────────────────────────────────┐
│ [Claude 레이어]  (비결정적·맥락 기반)                          │
│  · 자연어 요청 해석 → 스택/빌드도구 감지 → 프로파일 선택         │
│  · 통합 SARIF 수신 → 트리아지(TP/FP) · 도달성 보강 판정          │
│  · 사람이 읽는 보고서 작성(CWE 설명, 수정/업그레이드 가이드)      │
└───────────────▲──────────────────────────┬───────────────────┘
        통합 SARIF + 코드 컨텍스트           │ 프로파일·대상 지정으로 CLI 호출
┌───────────────┴──────────────────────────▼───────────────────┐
│ [오케스트레이터 CLI]  (결정적·재현가능)                         │
│  · 프로파일대로 스캐너 병렬 실행 · 타임아웃/에러 격리            │
└───────────────────────────┬──────────────────────────────────┘
                            │ 원시 출력(JSON/SARIF/텍스트)
┌───────────────────────────▼──────────────────────────────────┐
│ [정규화 레이어]  ← 이 프로젝트의 심장                           │
│  · 각 툴 출력 → SARIF 2.1.0 통일 → 중복제거 → 합의 스코어링      │
│  · CWE/OWASP 매핑 → severity 정렬 → baseline/VEX 억제 적용      │
└───────────────────────────┬──────────────────────────────────┘
                            │
┌───────────────────────────▼──────────────────────────────────┐
│ [스캐너 어댑터]  (각 1책임, 교체가능 인터페이스)                 │
│  Semgrep · Trivy · osv-scanner · Gitleaks · TruffleHog          │
│  (deep) SpotBugs+FindSecBugs · (eval) OWASP dep-scan            │
└───────────────────────────────────────────────────────────────┘
```

### 데이터 흐름
1. 사용자가 자연어로 점검 요청 → Claude가 프로젝트 스택/빌드도구 감지, 프로파일 결정
2. Claude가 오케스트레이터 CLI를 `scan --profile <p> --target <path>`로 호출
3. CLI가 어댑터들을 병렬 실행 → 각 원시 출력 수집
4. 정규화 레이어가 통합 SARIF 산출 (중복제거·합의·억제 적용)
5. Claude가 통합 SARIF + 실제 코드를 읽어 트리아지/도달성 판정
6. Claude가 Markdown 보고서 생성, 판정 결과를 VEX/baseline에 기록(억제 루프)

## 4. 컴포넌트별 책임·인터페이스·의존성

각 컴포넌트는 단일 책임을 가지며, 다른 컴포넌트의 내부를 모른 채 인터페이스로만 소통한다.

### 4.1 스캐너 어댑터 (`adapters/`)
- **책임**: 하나의 스캐너를 실행하고 원시 출력을 반환. 스캐너별 인자·환경 차이를 흡수.
- **인터페이스**: `run(target, options) -> RawResult{tool, format, payload, exit_code, stderr}`
- **의존성**: 해당 스캐너 바이너리(로컬 설치 또는 Docker 이미지)
- **불변식**: 어댑터는 정규화를 하지 않는다(원시 출력만). 새 스캐너 추가 = 어댑터 1개 추가, 코어 무변경.

### 4.2 오케스트레이터 (`orchestrator.py`)
- **책임**: 프로파일이 지정한 어댑터들을 병렬 실행, 타임아웃·실패 격리(한 스캐너 실패가 전체를 죽이지 않음).
- **인터페이스**: `scan(profile, target) -> list[RawResult]`
- **의존성**: 어댑터들, 프로파일 정의

### 4.3 정규화 레이어 (`normalize/`)
- **책임**: 원시 출력 → 통합 SARIF. 중복제거, 합의 스코어링, CWE/OWASP 매핑, severity 정렬, baseline/VEX 억제.
- **인터페이스**: `normalize(list[RawResult]) -> Sarif`, `suppress(Sarif, baseline, vex) -> Sarif`
- **의존성**: 없음(순수 변환). → **단위 테스트의 1순위 대상**.
- **불변식**: 입력 RawResult가 같으면 출력 SARIF가 같다(결정적).

### 4.4 프로파일 (`profiles.yaml`)
- **책임**: "수준·범위"를 선언적으로 정의(어떤 어댑터를 어떤 옵션으로).
- 아래 6절 참조.

### 4.5 Claude 레이어 (스킬/프롬프트 + 코드 질의 도구)
- **책임**: 프로파일 선택, 트리아지, 도달성 보강, 보고서.
- **코드 질의 인터페이스**: 기본 `grep`/`ast-grep`(구조적·결정적 매칭). 나중에 LSP(Serena) 어댑터로 교체 가능하도록 추상화.
- **불변식**: 도달성 판정은 시맨틱 유사도가 아니라 **정확한 호출부 매칭**으로 한다(false negative 방지).

### 4.6 보고서 생성기 (`report/`)
- **책임**: 통합 SARIF + Claude 트리아지 → Markdown 보고서.
- **인터페이스**: `render(Sarif, triage) -> Markdown`

## 5. 사용 시나리오 → 코어 1개 + 얇은 트리거 3개

세 시나리오는 별개 시스템이 아니라 **같은 코어 엔진의 다른 호출**이다. 코어를 먼저 만들고 트리거는 얇은 래퍼로 얹는다.

| 시나리오 | 구현 | 단계 |
|---|---|---|
| 프로젝트 풀스캔 (on-demand) | 코어 그대로 | MVP |
| 신규 라이브러리 패치 체크 | 코어의 **SCA 모듈만** 변경된 의존성에 (`quick` 프로파일) | MVP |
| 주기 점검 | 풀스캔을 launchd/cron + 직전 결과와 diff | 백로그 |

## 8. 정확도 설계 (4대 레버)

정확도는 "툴 추가"가 아니라 아래 레버에서 나온다.

| 레버 | MVP 적용 | 비고 |
|---|---|---|
| **① 도달성** (있다 ≠ 호출됨) | SCA: **OWASP dep-scan/atom** 1차 도달성(M4에서 평가 도입) + Claude 코드확인 보강. SAST: Semgrep taint(얕음 — 8.1 참조) | **JVM SCA 노이즈의 최대 감소원** |
| **② 분석 깊이** | `deep`에서 SpotBugs(바이트코드). 백로그: Opengrep/CodeQL | 패턴<데이터플로우<바이트코드 |
| **③ 다중엔진 합의** | Trivy ∪ osv (advisory 커버리지↑, FN↓), 같은 위치 중복 → confidence 가중 | 정규화 레이어가 수행 |
| **④ 메타 트리아지 + 억제** | Claude가 코드 컨텍스트로 TP/FP 판정 → **VEX/baseline 기록 → 재발 억제** | precision 최대 레버, 노이즈 단조감소 |

### 8.1 무료 taint의 한계 (중요)
Semgrep CE의 taint는 **단일 함수·단일 파일 내(intrafile)만** 추적한다. 실무 가치가 큰 cross-function/cross-file taint는 Pro 전용이다. 따라서 "taint로 공짜 정확도"를 과신하지 않는다. 깊은 데이터플로우가 필요하면 백로그의 **Opengrep**(해당 기능을 OSS로 복원) 또는 **CodeQL**(OSS 한정 무료)을 평가한다.

### 8.2 정확도 측정
OWASP Benchmark는 v1.2beta(2016)로 동결·합성데이터라 현대 SAST 평가엔 부적합. 대신 **의도적 취약 픽스처 프로젝트**(알려진 CVE 라이브러리, 하드코딩 시크릿, 취약 코드 패턴)로 precision/recall을 회귀 측정한다(11절).

## 9. 흔히 깨지는 지점 & 설계적 대응

| 실패 모드 | 왜 죽나 | 대응 |
|---|---|---|

| 도달성 과신 | 동적 디스패치/리플렉션은 정적 분석 사각 | 사각지대를 보고서에 명시, 수동 확인 표기 |

## 10. 기술 선택

- **오케스트레이터/정규화 구현 언어**: **Python 권장**. 근거 — 스캐너 대부분이 CLI라 subprocess 글루에 적합, SARIF 라이브러리 풍부, 병렬 실행·빠른 반복 용이. *(단일 바이너리 배포가 중요해지는 CI 단계에선 Go 재검토. 사용자가 JVM 일관성을 원하면 Kotlin도 가능 — plan에서 확정)*
- **스캐너 실행 방식**: 로컬 바이너리 우선, 버전 고정이 필요하면 Docker 이미지 옵션.
- **통합 포맷**: SARIF 2.1.0 (기계용) + Markdown (사람용).
- **억제 포맷**: OpenVEX / SARIF baseline (표준·git 추적·감사가능).

## 11. 테스트·검증 전략

- **정규화 레이어 골든 테스트**: 각 툴의 샘플 원시 출력 → 기대 통합 SARIF. 결정성 보장.
- **취약 픽스처 프로젝트**: 알려진 CVE 의존성 + 하드코딩 시크릿 + 취약 코드 패턴을 심은 소형 JVM 프로젝트. E2E 탐지 검증.
- **정확도 회귀**: 픽스처의 알려진 TP/FP로 precision/recall 추적(8.2).
- **어댑터 스모크 테스트**: 각 스캐너 설치 여부·기본 실행(`doctor` 커맨드).

## 12. 마일스톤

| 단계 | 산출물 | 검증 |
|---|---|---|
| **M0 스캐폴딩** | CLI 뼈대, 프로파일 정의, `doctor`(스캐너 설치 점검) | 스캐너 감지됨 |
| **M1 SCA 수직슬라이스** | Trivy+osv 실행→정규화→통합 SARIF→Claude 트리아지→Markdown | 픽스처 CVE 탐지·보고 |
| **M2 Secret** | Gitleaks+TruffleHog 어댑터 추가 | 하드코딩 시크릿 탐지·검증 |
| **M3 SAST** | Semgrep 어댑터 추가 | 취약 패턴 탐지 |
| **M4 정확도** | Trivy∪osv 합의 dedup, VEX/baseline 억제, dep-scan 도달성(평가) | FP 억제·재현성 확인 |
| **M5 deep** | SpotBugs+FindSecBugs(빌드 연동) | 바이트코드 탐지 |

M1을 SCA로 잡는 이유: 전체 파이프라인(실행→정규화→통합→트리아지→보고서)을 가장 작은 수직 슬라이스로 검증하며, SCA가 결정적이고 "라이브러리 체크" 시나리오를 즉시 충족한다.

## 13. 라이선스 노트

현재 **개인 사용 단계**라 모든 채택 도구의 라이선스 제약은 무관하다. **사내/상용 코드베이스로 확장 시** 재검토 대상:
- **TruffleHog (AGPL-3.0)**: 통합 방식에 따라 민감 → 검증 단계를 별도 프로세스로 격리 실행하는 방안 검토.
- **CodeQL**: 사유 코드 분석은 유료 라이선스 필요.
